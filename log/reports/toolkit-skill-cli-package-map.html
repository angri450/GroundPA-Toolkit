一句话结论

现在的结构只是“半对齐”，还不是你设想的那种一伸手就能顺着胳膊摸到骨头的结构。

大多数 skill 到 CLI 这一层已经差不多对上了，但 CLI 到包层、包到运行时层、策略 skill 到执行 skill 的边界 还没有统一标出来。最明显的乱点就是 OCR 线。

先把四层说人话

1. Skill 层

这是给 AI 看的入口词。它应该尽量贴近用户说法，让模型一看到请求就能想到该触发谁。

2. CLI 层

这是确定性执行入口。最好和 skill 同名，或者至少是一眼能对应上的缩写/别名。

3. 包层

这是代码归属边界。它不一定要和 skill/CLI 完全同名，但必须能稳定映射，不要每次都靠猜。

4. 运行时包/外部仓库层

只在 OCR、PDF render 这种重依赖场景特别重要。这里不是 AI 主入口，但必须在映射图里标出来，不然一出安装问题就找不到源头。

哪一层必须同名，哪一层不用死磕同名

层之间最好状态为什么
skill ↔ CLI 尽量同名或极易联想 这是 AI 路由最关键的一跳。这里越省翻译，误触发越少。
CLI ↔ 核心包 不必强行同名，但必须有固定映射 包层服务的是代码组织，不一定和用户语言一模一样。比如 `word` 对应 `Docx` 就能接受，只要表是清楚的。
核心包 ↔ 运行时包 只在重依赖场景显式标出 不是每个 skill 都有 runtime 仓库，但 OCR 这类必须写清楚，不然排障会乱。

最乱的一条：OCR 线

你说得对，这条线现在最需要标清。

仓库名是 `Nong.OcrRuntime`，不是别的。 但当前实际层次是：

multimodal skill -> nong ocr CLI -> Angri450.Nong.MultiModal 核心包 -> Nong.OcrRuntime 仓库 -> Angri450.Nong.OcrRuntime.* 五个平台运行时包

层当前名字问题建议
Skill multimodal 太泛，像什么图像、多模态、OCR 都能装进去。 优先改成 ocr。这里不建议再绕成 ocr-routing，因为 CLI 已经叫 ocr 了。
CLI nong ocr 这一层名字反而是对的。 保持不动，作为执行主轴。
核心包 Angri450.Nong.MultiModal 代码包名保留了历史“多模态”说法。 短期不用急着改包名，但要在映射表里明确写：ocr 命令组的核心包就是它。
运行时仓库 Nong.OcrRuntime 仓库名、包名、skill 名当前不在一条线上。 把仓库名单独标出，告诉维护者：本地 OCR runtime 在这里维护。
运行时包 Angri450.Nong.OcrRuntime.WinX64 等 5 个包 安装失败时很容易只看到 `ocr install-model`，却不知道背后是哪些包。 在 skill 文档和对照表里固定写出 5 个 RID 包，排障就有抓手。

OCR 这条线其实已经拆好的 reference

这里有一个要纠偏的地方：OCR skill 的下钻材料其实已经拆了，不是没拆。

当前仓库里已经有：

  • multimodal/references/ocr-local.md：本地 OCR
  • multimodal/references/ocr-cloud.md：云端 OCR / to-word
  • multimodal/references/image-analyzer.md：图像结构 QA

所以 OCR 现在的主要问题不是“没有分层”，而是外层 skill 名和内部三条路、CLI 命令组、runtime 仓库名没有串成一根主轴。

所以你设想的“像胳膊一样指挥”，和现在一样吗

不一样。现在只是“有骨头，但骨节没有都露出来”。

现在最接近你设想的是 `word/pdf/excel/chart/diagram/pptx/inspect` 这批。最不像的是 `multimodal`，其次是 `skill-manager`、`progress-report`、`slice`、`icons` 这些需要解释一层才能懂的点。

哪个更方便

对 AI 和维护者来说，你设想的结构更方便。

但要稍微收一刀：不是要求四层所有名字机械完全一样，而是要求 一条固定主轴 + 一张明确映射表。

  • Skill 和 CLI：越同名越好，这是 AI 路由主轴。
  • CLI 和核心包：可以不同名，但必须稳定映射。
  • 核心包和 runtime 包：只在 OCR 这类重依赖路径显式标出。

我给你的落地建议

优先级动作理由
P1 把 Toolkit 的 multimodal skill 定位改成 ocr 让 skill -> CLI 这一跳直接对齐 `nong ocr`。
P1 在 README / 审计页 / skill 文档里固定写出 OCR 链：ocr -> nong ocr -> Angri450.Nong.MultiModal -> Nong.OcrRuntime -> Angri450.Nong.OcrRuntime.* 一旦装不上、跑不动、镜像不同步，就知道该去哪一层看。
P2 给 slice、icons、skill、progress-report 都补“核心包是谁”的说明 这些现在都要读过代码才知道后面是谁。
P2 把 `skill-manager` 明确声明为治理 skill，不再伪装成执行 skill 否则它会持续和 `nong skill` 抢入口。
