一句话结论

Toolkit 现在不是“整体都废了”，而是“有一批已经成型，一批偏薄，一批名字和触发把自己搞糊了”。

最严重的问题不在命令少，而在 skill 的命名、description、references、examples、失败回收闭环没有按同一标准建设。结果就是：有的 skill 很能用，有的 skill 只有门牌，没有房间；有的 skill 明明对应一个 CLI 命令组，却起了一个太虚的名字，AI 很难稳稳触发到对的入口。

先说人话：现在到底是什么局面

A 类：骨架已经稳

word、pdf、literature、chart、excel、diagram

这些 skill 已经像工具手册了，边界大体清楚，命令面能对得上 CLI，主要问题是还缺例子和失败样本。

B 类：方向对，但偏薄

skill、slice、pptx、genre、inspect

这些 skill 并不是错，只是有的太短，有的材料不够分层，有的名字不够直观，有的把两个场景写在一个入口里。

C 类：现在真不好用

skill-manager、multimodal、progress-report、icons

这四个最该先动。不是都要推倒重来，但至少要重写 description、补 references，个别还要改名或拆层。

系统级问题

问题现在是什么样为什么会出事建议
命名标准不统一 有的 skill 直接跟 CLI 命令组同名，比如 word、pdf；有的却用了更虚的词，比如 multimodal。 AI 看到需求时，会更容易把问题往宽泛词上靠，导致误触发。 能镜像 CLI 的，就尽量镜像 CLI；如果不是 CLI 镜像，而是策略或工作流 skill，就用角色名。
渐进式披露深度不均匀 word 有 7 份 reference，skill-manager、progress-report、icons 这些却只有一个主文件甚至没有下钻材料。 有的 skill 触发后能继续往下读，有的只看到一段总说明，后面没有路。 高频 skill 至少 2 份 reference；策略 skill 至少 3 份 reference；简单命令镜像 skill 也要有最少 1 份边界说明。
例子纪律几乎没立起来 15 个 skill 里只有 diagram 真正带了 examples，其他 14 个都没有。 没有例子，skill 容易停留在“口头上会”，遇到真实请求就会飘。 把“每个 skill 至少 1 个成功例子，复杂 skill 再加 1 个失败例子”写成硬规则。
失败反馈闭环没有落在 skill 结构里 仓库里有 log/debug，也有用户反馈，但大多数 skill 没有明确告诉 AI 如何把失败沉淀回 reference 或例子。 技能会越改越散，重复踩坑，触发和边界问题会反复出现。 给策略类 skill 单独写 feedback/regression reference，把真实失误回收到例子和边界里。
有些 skill 把不同层次混写 比如 inspect 同时写诊断和生成，progress-report 同时写报告生成和旧工具迁移政策。 AI 会不知道这是“执行动作”还是“治理动作”。 执行入口和治理规则分开；主 skill 只做分流，细节放 reference。

最该先处理的四个 skill

Skill问题说人话根因优先动作
skill-manager 像“凡是跟 skill 有点关系都能来”的总收发室。 名字太大，description 太宽，0 references，0 examples，策略和执行混在一起。 先收窄 description；至少补 authoring / trigger-audit / feedback-loop 三份 reference；长期拆成策略入口加子 skill。
multimodal 明明实际在教 nong ocr，却起了一个太泛的名字。 外层 skill 名和 CLI 命令组不对齐；下层其实已经有 ocr-local.md、ocr-cloud.md、image-analyzer.md 三份 reference。 优先改名到 ocr；把现有三条 reference 在总入口、README 和审计页里标清楚，再把 Nong.OcrRuntime 仓库链路写明。
progress-report 方向没错，但现在像一份半成品工作说明书。 0 references，0 examples，同时承担日志结构说明和“旧工具别再乱放”的治理口径。 补 2 份 reference：日志结构与报告生成；迁移策略单独放边界段或独立 reference。
icons 太薄了，看起来像命令速查表，不像可判断边界的 skill。 0 references，0 examples，名字又容易让 AI 联想到通用图标设计，而不仅是 Bioicons 检索。 如果只做 Bioicons，考虑名字或 description 明确成 bioicons / scientific icon search；至少补 1 份 reference 说明范围和不做什么。

每个 skill 到底该怎么判

Skill定位触发精度分层深度例子主要问题建议
word CLI 镜像 + 高复杂工作流 高 厚 无 内容很全，但还没有把真实成功/失败案例固化。 保持结构，不要大拆；补学术格式、表格重排、旧 DOCX 修复各 1 个例子。
pdf CLI 镜像 + 证据型读取 高 中 无 主体清楚，但 reference 还不够厚，例子为零。 补文本层 PDF、扫描 PDF、混合 PDF 三种判路例子。
literature CLI 镜像 + Provider 路由 高 中 无 Provider 边界讲清了，但例子还是不够。 补 1 个 DOI 查找例子、1 个复杂检索式例子、1 个 provider 失败例子。
chart CLI 镜像 + 数据到图 高 中 无 命令面清楚，但缺“输入坏了怎么修”的例子。 补分组 JSON、折线 spec、显著性展示失败案例。
excel CLI 镜像 + 数据准备 高 够用 无 整体清楚，但缺“从 Excel 到 chart”的通路例子。 补 1 个从工作表到 groups.json 再到 chart 的端到端例子。
diagram CLI 镜像 + 图形生成 高 够用 有 目前是全仓最像“完成态”的一个 skill。 可作为 examples 规范样板，反向指导别的 skill。
skill CLI 镜像 + 生命周期关卡 高 薄 无 边界清楚，但 0 references，0 examples。 补 1 份 packaging gate reference，至少给 validate / scan / package 一套最小顺序例子。
slice CLI 镜像 + 通用包读取 高 薄 无 本体短而清楚，但缺块级读取例子和严格证据例子。 补 1 份 reference：block/asset/read-order 的典型问答。
pptx CLI 镜像 + 只读入口 高 薄 无 读得懂，但太短，用户容易误以为支持编辑。 补 1 个“只读，不改稿”的 reference，明确和 future write path 的边界。
genre CLI 镜像，但名字偏抽象 中 薄 无 genre 这个词不够直观，AI 读到“模板”时未必先想到它。 保留 CLI 对齐的同时，在 description 里更强地写 template catalog / preset discovery；必要时考虑命令别名。
inspect CLI 镜像 + 论文/公文语义工作流 中 偏薄 无 诊断和起草都放在一个入口里，方向对，但层次还不够清楚。 拆成诊断 reference 和写作 reference；给论文诊断、公文起草各 1 个例子。
multimodal 本质是 OCR 路由 低 已有分层 无 真正问题是外层名字太虚，不是内部 reference 没拆。 优先改名到 ocr，并把 ocr-local / ocr-cloud / image-analyzer 三条已存在分路标到总入口。
progress-report CLI 镜像 + 日志治理 中 薄 无 报告生成和治理规则写在一起，没有下钻材料。 至少补日志结构 reference 和报告生成 reference。
icons CLI 镜像，但范围表达过宽 中 薄 无 太像“所有图标问题都归我”，其实它只是 Bioicons 检索。 补边界说明，必要时把名字朝 Bioicons 靠。
skill-manager 策略治理 skill 低 薄 无 总兜底入口，什么都想管。 优先级最高，先收窄，再拆层。

关于名字：哪些词 AI 更容易看懂

你前面一直强调“名字得让 AI 看得懂，词得对得住意思”，这个判断是对的。

Toolkit 里最典型的名字问题，不是所有 skill 都有，而是集中在少数几个词太虚、太泛、太不像 CLI 命令组。

当前名字问题更稳的方向说明
multimodal 太大，像什么图片、OCR、视觉、多模态都能进。 ocr 它实际就是围绕 nong ocr 的路由 skill。内部三份 reference 已经分好，缺的是总入口名字和 Nong.OcrRuntime 链路说明。
skill-manager 像总后台，天然吸走过多问题。 skill-policy，或保留原名但大幅收窄 description 如果它保留总入口身份，就要显式声明“我只做策略分流”。
icons 太像通用图标设计，而非 Bioicons 检索。 bioicons 或在 description 中明确 “scientific icon search” 如果 CLI 就叫 icons，也可以不改命令名，但 skill 的描述必须更准。
genre 词本身偏文学/体裁，和“模板目录”不是一眼对齐。 保留命令名，强化 description 中的 template/preset 词 这里不一定要真改名，但至少要把触发词写得更像“模板库”。

最终应该长成什么样

1. 命令镜像 skill

  • 名字尽量跟 CLI 命令组一致：word、pdf、excel、pptx、slice、skill、chart、diagram、icons、literature
  • 主文件负责分流、边界、命令面
  • reference 负责细场景
  • 至少一个成功例子

2. 工作流 skill

  • 处理跨命令组合，但名字要说清是“什么工作流”
  • inspect 属于论文/公文工作流
  • ocr 属于 OCR 路由工作流，不该继续挂在笼统的 multimodal 入口下
  • 要把不同子路径拆成 reference

3. 策略治理 skill

  • 不直接承接所有执行细节
  • skill-manager 只该做策略、分流、质量约束
  • progress-report 要么是纯报告生成 skill，要么是带少量治理边界，但不能混成一团

4. 统一质量底线

  • 每个 skill 都要有清晰边界
  • 每个复杂 skill 都要有 references
  • 每个 skill 至少 1 个成功例子
  • 每个高风险 skill 还要有 1 个失败例子或回归案例

建议的整改顺序

优先级动作理由
P1 重写 skill-manager description，并补至少 3 份 reference 这是当前最明显的误触发源。
P1 把 multimodal 改到和 nong ocr 直接对齐，并把现有三份 OCR reference 与 Nong.OcrRuntime 链路标清 问题主要在外层名和映射断裂，不在下层材料缺失。
P1 给 progress-report 和 icons 补 references 与边界说明 这两个不是理念错，是太薄。
P2 给 inspect、slice、skill、pptx、genre 补最小例子和 reference 把“可用但偏薄”的 skill 补成“可稳定使用”。
P2 给 word、pdf、literature、chart、excel 建失败样本 这些 skill 已经够强，下一步该补闭环，而不是继续堆规则。
P3 形成统一 skill 模板：命名规则、description 模板、reference 最低数、example 最低数 不立模板，后面还会继续出现“有人写得厚，有人只写了个门牌”的情况。

最后一句判断

Toolkit 的问题不是“没有想法”，而是“好想法没有被做成统一的 skill 工程规范”。

skill-manager 只是最显眼的一坨；真正要修的是整套 skill 系统的命名标准、分层深度、例子纪律和失败闭环。

但也别一刀切地全盘重写。该保留的保留，该补材料的补材料，该改名的改名，该拆层的拆层。
